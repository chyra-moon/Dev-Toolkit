# JSON 格式比对对比特性需求书

此文件用于锁定和统一定义项目中两个 JSON 字符串文本内容在发生视觉差异对比（Diff）时，必须通过技术途径实现的几个极端体验指标和排版原则。

## 1. 过滤：空行静默与降噪机制
算法绝不能把 JSON 中的纯空行加入作为“被删除”、“被增加”的混分误判因子。算法设计必须以某种安全的容错解析（或者正则前置清洗）方式，**绝对忽略掉所有的纯空行元素**。

## 2. 染色：背景色显性着色（底线要求）
无论任何组件渲染架构，页面只要有改动必须要在具体的代码列上输出具有一定透明度的整行背景色填充。此项需求不仅限于文字标注！
- **内容发生更替 (Modified)**：要求渲染出明亮的底色，当前采用淡黄色。（如 `rgba(202, 138, 4, 0.15)`）
- **有全新的项目增加 (Added)**：无论出现在结尾还是穿插在中间，当前要求纯系果绿色底。（如 `rgba(34, 197, 94, 0.15)`） 
- **原有项目被剥落去除 (Removed)**：要求红底呈现警告感知。（如 `rgba(239, 68, 68, 0.15)`）

## 3. 排版：绝对禁止的“阶梯式”差位渲染
这是一条极其核心的视觉要求：**只要发生了针对同一目标区块/对象的【非分离式联合删改】（这里少了一块，右侧同样的位置新出现另外一块），这部分必须在同一个行视距上，实现“脸贴脸”级别的起手水平对齐。**

绝对不允许发生因为原生 Diff 工具按先删后补的顺序输出结果，导致在左上角挖掉一块红底之后，对应的全新增加代码掉落在偏右下角处，构成肉眼看着撕裂的**“斜角阶梯”级渲染错误**的情况。

- 如果两侧对比行数长短不一：较短的那一侧区域（短多少行）必须在这整块连续对比区的**正下方**，塞入一个等身高度的空气填充占位块 `Spacer`。用来强行把代码的根基托稳垫平，最终使得两侧在退出此局部区域恢复到未改变部分时，再次恢复彻底的水平对齐展示。

## 4. 树形乱序规范化 (预处理要求)
对比前必须在后台将乱序的 JSON 进行**以键名字典序 (A-Z)** 为基准的强制排序格式化。这能避免逻辑相同但位置颠倒的 JSON 直接触发大面无意义的增删报错，暴露出真正修改的数据。

## 5. 强类型敏感性
本工具对比数据类型极度敏感。例如原值是数字 1，新值是字符串 '1'，或是 1.0 变为了 1，都必须严格判定为被修改（属于黄色 Modified），需要明确标注出来，绝不能被算法私自兼容和蒙混过关。

## 6. 键值对变动法则与精细化追踪
这是一个核心的 Diff 逻辑判定边界：
- **键（Key）不同**：如果同一层级下的 Key 发生了变化，必须被判定为对旧 Key 的**完全删除（红色）**和对新 Key 的**完全新增（绿色）**，不能将 Key 的变化当作修改。
- **键相同、值（Value）不同**：只有当 Key 严格相等，但其对应的 Value 发生变化时，才被判定为**被修改（黄色 Modified）**。在这种情况下：
  1. 左右两边的对应行都必须将背景标为黄色。
  2. 必须进一步剖析行内变动，将**行内具体变动的值（无论是哪个字、数值或单词）**单独用颜色更深的高亮块圈出来，或使用左侧删除线/右侧加粗的方式凸显。

## 7. 差异节点标注（右侧视觉简化）
针对整体的视觉呈现：
- **行状态指示**：在变更代码的行号区域上，做点状等特殊标记。
- **视觉徽章**：在代码右侧合适的位置显示对应状态的小圆点/色块徽章（例如提示该层次或行为新增/修改/删除的徽章点）。**此徽章仅在右侧展示。**
- **核心差异框体**：直接把右侧发生变化（增、删、改）的整个内容区块用边框框起来进行视觉隔离隔离。

## 8. 默认层级折叠与镜像同步
- **初始折叠态**：对比完成后，左右两侧 JSON 树默认折叠呈现**第一级（Level 1）**状态。使用者可通过上述的徽章圆点一眼看出哪一级的折叠内部含有差异。
- **动作双边同步**：在对比状态下，左右侧面板的操作完全绑定统一。无论是通过鼠标点击折叠图标，还是使用快捷键进行【展开/折叠】任何层级，**两边都必须严格同时、同层级地发生完全镜像的动作**，实现视野的绝对同步连动。
